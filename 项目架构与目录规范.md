# 项目架构与目录规范

## 核心原则

- 80/20：先做当前真正需要的，不搞过度设计
- 简单直接：不搞复杂统一入口，每个方案独立完整
- 层级清晰：01-physical → 02-platforms → 03-infrastructure → 04-applications

## K8s 应用目录规范

### 目录结构
```
<component-dir>/
├── manifests/                      # K8s 原生清单（可选）
│   ├── remote-<name>-<version>.yaml    # 远程下载的原始文件，用 remote- 前缀
│   ├── <config-name>.yaml              # 自己写的配置文件，带详细注释
│   └── ...
├── helm/                           # Helm Chart（可选）
│   ├── remote-<chart-name>         # 远程下载的 Chart，用 remote-前缀，这是一个文件夹，相当于 helm pull xxx --version xxx --untar
│   ├── values-prod.yaml            # Prod 环境 Values, 包含持久化(if)
│   └── ...
├── operator/                       # operator（可选）
│   └── ...
├── kustomize/                      # Kustomize（可选）
│   ├── base/
│   └── ...
├── install.sh                      # 安装脚本（只有一个，默认方式）
├── uninstall.sh                    # 卸载脚本（只有一个）
├── README.md                       # 简单说明：只讲怎么用
└── 高级配置指南.md                 # 详细高级玩法（可选）
```

### 命名约定

#### 远程资源
- 格式：`remote-<name>-<version>.yaml`
- 示例：`remote-metallb-v0.14.8.yaml`、`remote-ingress-nginx-v1.12.0.yaml`

#### 配置文件
- 自己写的配置文件，用有意义的名字
- 示例：`ipaddresspool.yaml`、`l2advertisement.yaml`
- **必须带详细注释**，说明配置项含义

#### 环境区分
- 大多数应用不需要区分 dev/prod，直接一个配置文件
- 确需区分的，通过文件名后缀：`values-dev.yaml`、`values-prod.yaml`

#### 脚本要求
- 简洁为主，允许任意目录直接sh执行，需要处理脚本目录
- 应用根目录下的安装脚本是推荐安装方式，社区推荐或者官方推荐
- 对于每一种安装方式里面也需要有最简洁的安装卸载脚本。

### README.md 要求

```

## Docker Compose 应用目录规范

### 目录结构
```
<app-dir>/
├── docker-compose.yaml            # 主要配置（可选 dev/prod 后缀）
├── docker-compose-dev.yaml        # Dev 环境（可选）
├── docker-compose-prod.yaml       # Prod 环境（可选）
├── config/                        # 配置文件（可选）
├── data/                          # 数据目录（gitignore）
├── install.sh                     # 安装脚本（可选）
├── uninstall.sh                   # 卸载脚本（可选）
└── README.md                      # 简单说明：只讲怎么用
```

### README.md 要求

**只讲怎么用，不列出文件树**，简单为主：
```
# <应用名>

## 快速开始
```bash
docker-compose up -d
```

## 验证
```bash
docker-compose ps
```

## 停止
```bash
docker-compose down
```
## 域名规范
