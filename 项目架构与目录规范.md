# 项目架构与目录规范

## 核心原则

- 80/20：先做当前真正需要的，不搞过度设计
- 简单直接：不搞复杂统一入口，每个方案独立完整
- 层级清晰：01-physical → 02-platforms → 03-infrastructure → 04-applications


## 目录结构

```
<project-root>/
├── 01-physical/                    # 物理层：裸机、基础系统配置
├── 02-platforms/                   # 平台层：Docker、K3s、Kubernetes

├── 04-apps-docker/                 # 业务应用 Docker Compose # 业务方开发的应用，对外提供功能服务
├── 04-apps-k8s/                    # 业务应用 K8s # 业务方开发的应用，对外提供功能服务 # 如：demo-go-tiny

## K8s 应用目录规范

### 目录结构
```
<component-dir>/
├── manifests/                      # K8s 原生清单（可选）
│   ├── remote-<name>-<version>.yaml    # 远程下载的原始文件，用 remote- 前缀，此文件夹禁止修改
│   ├── <config-name>.yaml              # 自己写的配置文件，带详细注释
│   └── ...
├── helm/                           # Helm Chart（可选）
│   ├── remote-<chart-name>         # Helm方式部署必须离线保留 Chart 需要，用 remote-前缀，这是一个文件夹，相当于 helm pull xxx --version xxx --untar，或者是 github 上托管的 chart 仓库内容 . 此文件夹禁止修改
│   ├── values-prod.yaml            # Prod 环境 Values, 包含持久化(if)
│   └── ...
├── operator/                       # operator（可选）
│   └── ...
├── kustomize/                      # Kustomize（可选）
│   ├── base/
│   └── ...
├── install.sh                      # 安装脚本（只有一个，默认方式）
├── uninstall.sh                    # 卸载脚本（只有一个）

├── 高级配置指南.md                 # 详细高级玩法（可选）
└── 踩坑记录.md                     # 踩坑记录：部署过程中遇到的问题和解决（可选）
```

### 命名约定

#### 远程资源
- 格式：`remote-<name>-<version>.yaml`
- 示例：`remote-metallb-v0.14.8.yaml`、`remote-ingress-nginx-v1.12.0.yaml`
- 远程资源禁止修改

#### 配置文件
- 自己写的配置文件，用有意义的名字
- 示例：`ipaddresspool.yaml`、`l2advertisement.yaml`
- **必须带详细注释**，说明配置项含义

#### 环境区分
- 大多数应用不需要区分 dev/prod，直接一个配置文件
- 确需区分的，通过文件名后缀：`values-dev.yaml`、`values-prod.yaml`

#### Helm README.md 要求

每个 helm/ 目录必须包含 README.md，说明如何使用本地离线 Chart 部署。
内容模板（`helm/README.md`）：

~~~markdown
# Helm Chart: <名称>

## 来源

- **Chart**: <chart-name> <version>
- **远程仓库**: <仓库URL>
- **下载方式**:
  ```bash
  helm repo add <repo> <repo-url>
  helm pull <repo>/<chart> --version <version> --untar
  mv <chart> remote-<chart>-<version>
  ```
- **离线路径**: `remote-<chart>-<version>/`（禁止修改此目录）

## 本地安装

```bash
# 从本地离线 Chart 安装（使用预配置的 prod 环境 values）
helm upgrade --install <release> ./remote-<chart>-<version> \
  -n <namespace> --create-namespace \
  -f ./values-prod.yaml

# 仅查看默认 values
helm show values ./remote-<chart>-<version>
```

## 目录说明

| 文件/目录 | 说明 |
|-----------|------|
| `remote-<chart>-<version>/` | 离线 Chart 文件，helm pull 原样保留，禁止修改 |
| `values-prod.yaml` | 生产环境配置覆盖，按环境文件后缀区分 |
| `values.yaml`（可选） | 自定义 Chart 的默认 values（非 remote 方式时使用） |

## 关键配置

列出主要配置项，如持久化、Ingress、资源限制等。
可以在 values-prod.yaml 中查看完整配置。

## 升级

```bash
# 替换 remote-xxx 为新版后
helm upgrade <release> ./remote-<chart>-<version> -n <namespace> -f ./values-prod.yaml
```

## 卸载

```bash
helm uninstall <release> -n <namespace>
```
~~~

#### 脚本要求
- 简洁为主，允许任意目录直接sh执行，需要处理脚本目录
- 应用根目录下的安装脚本是推荐安装方式，社区推荐或者官方推荐
- 对于每一种安装方式里面也需要有最简洁的安装卸载脚本。

## Docker Compose 应用目录规范

```

├── docker-compose.yaml            # 主要配置（可选 dev/prod 后缀）
├── docker-compose-dev.yaml        # Dev 环境（可选）

├── uninstall.sh                   # 卸载脚本（可选）
└── README.md                      # 简单说明：只讲怎么用
```

### README.md 要求

**只讲怎么用，不列出文件树**，简单为主，包括记录踩坑，容易遇到的问题：

## 快速开始
```bash
docker-compose up -d
```

## 验证
```bash
docker-compose ps

## 停止
```bash
docker-compose down
```
## 域名规范

- 避免使用 `*.local` 域名（macOS 特殊处理，会有 5 秒延迟）
- 统一使用 `*.czw-sre.internal` 域名


## Git 提交规范

- 格式：`type(scope): subject`
- 示例：`feat(metallb): add advanced config guide`
- 在提交信息末尾，附工具版本号和使用的模型


## 其他约定

- 国内环境：GitHub 资源加 `https://gh-proxy.com/` 前缀
- 搜索：必应
- 后端框架：Go 项目用 Echo
- 已经配置过镜像源，对于镜像禁止采用 docker.io 前缀
