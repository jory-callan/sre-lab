# 踩坑记录

## 1. redis-operator sentinel 密码认证失败

### 现象

部署 Sentinel HA 后，sentinel Pod 正常启动但连不上 Redis 主库，日志：
```
+sdown master myMaster 10.42.1.15 6379
```

`SENTINEL masters` 显示 `status=sdown`。

### 根因

operator 的**独立 `RedisSentinel` CR 不支持密码认证**，它的 spec 中没有 `redisSecret` 字段。
查看源码 `internal/controller/redissentinel/`，独立 sentinel controller **不处理密码注入**。

密码注入逻辑在 `internal/controller/redisreplication/redisreplication_sentinel.go` 中，
它从 `RedisReplication.spec.kubernetesConfig.redisSecret` 读取密码，通过 `MASTER_PASSWORD` 环境变量传给 sentinel Pod。

### 正确做法

**不要创建独立的 `RedisSentinel` CR**，将 sentinel 配置内嵌在 `RedisReplication` CR 中：

```yaml
apiVersion: redis.redis.opstreelabs.in/v1beta2
kind: RedisReplication
spec:
  kubernetesConfig:
    redisSecret:                # ← sentinel 密码从此继承
      name: redis-auth
      key: password
  sentinel:                     # ← 内嵌，operator 自动创建 sentinel Pod
    size: 3
    image: quay.io/opstree/redis-sentinel:latest
    quorum: "2"                 # 注意值必须是字符串，不是数字！
```

operator 会自动创建 `redis-replication-s-0/1/2` 三个 sentinel Pod，
密码从 `kubernetesConfig.redisSecret` 自动注入。

### 字段类型陷阱

`sentinel` 中的 `quorum`、`parallelSyncs`、`failoverTimeout`、`downAfterMilliseconds`
**必须使用字符串类型**，写数字会 CRD 校验失败：

```
spec.sentinel.quorum: Invalid value: "integer": spec.sentinel.quorum in body must be of type string
```

### 验证

三个 sentinel Pod 都成功连接后，`SENTINEL masters` 输出 `status=ok`：
```
master0:name=mymaster,status=ok,address=10.42.1.20:6379,slaves=2,sentinels=3
```

## 2. redis-operator 必须使用定制镜像

### 现象

Redis 实例 Pod 启动正常，但密码认证失败：
```
AUTH failed: ERR AUTH <password> called without any password configured
```

### 根因

operator 的 `redisSecret` 密码认证机制需要 operator 自己的**定制镜像**（`quay.io/opstree/redis`），
这个镜像有自定义 entrypoint 负责密码注入、配置合并等 hook。

官方 `redis:7.2-alpine` 没有这些 hook，所以 `redisSecret` 不会生效。

### 正确做法

```yaml
kubernetesConfig:
  image: quay.io/opstree/redis:v7.0.15  # 必须用 operator 定制镜像
```

v7.0.x 仍为 BSD 开源协议（v7.4+ 改为 SSPL/RSAL），商用无风险。
