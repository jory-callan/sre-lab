# 踩坑记录

## 1. Percona Operator 不适合 standalone 单节点

### 现象

用 `clusterType: async` + `size: 1` 想跑单节点，结果：
- operator 强制要求 async 模式必须有 orchestrator + HAProxy
- 即使配了，healthcheck 因缺少 operator 用户导致 readiness 永远不过
- 整个 operator 的设计目标就是 HA 集群（group-replication 或 async + 3 节点），单点绕不过去

### 结论

standalone 用原生 manifests（kubectl apply），cluster 才用 Percona Operator。

## 2. Bitnami MySQL 社区镜像已下架

### 现象

```
helm pull bitnami/mysql
```
失败。Bitnami Helm Chart 已迁移到 OCI registry（registry-1.docker.io），国内无法访问。

更严重的是 Bitnami 在 Docker Hub 上的社区镜像已全部撤下，
`bitnami/mysql:8.4.5` 已不存在。

### 结论

不再使用 Bitnami 任何产品。standalone 直接使用 percona-server 官方镜像。

## 3. standalone StatefulSet 的注意事项

### readinessProbe 使用 mysqladmin ping

StatefulSet 中 readinessProbe 使用 `mysqladmin ping -uroot -p$(MYSQL_ROOT_PASSWORD)`，
依赖环境变量 `MYSQL_ROOT_PASSWORD` 从 Secret 中注入。

如果修改了密码但忘记更新 Secret，readiness 会持续失败。

### init 阶段慢

percona-server 镜像首次启动需要初始化 InnoDB 和系统表空间（约 10-30 秒），
readinessProbe 的 `initialDelaySeconds: 10` 可能不够，如果一直不 Ready 可以检查日志：

```bash
kubectl logs -n mysql mysql-0 | tail -20
```

正常启动最后一行应该是：
```
/usr/sbin/mysqld: ready for connections. Version: '8.4.8-8'  socket: '/var/lib/mysql/mysql.sock'  port: 3306
```
