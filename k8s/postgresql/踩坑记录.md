# 踩坑记录

## 1. CNPG 使用的镜像版本策略

### 说明

CloudNativePG 维护自己的 PostgreSQL 镜像（`ghcr.io/cloudnative-pg/postgresql`），
不要使用官方 `postgres:17` 镜像。

从 v1.24+ 开始，CNPG 弃用了裸版本标签（如 `17`、`latest`），
改为固定格式标签：

```
<PG版本>-<镜像类型>-<Debian版本>
```

例如：`17-minimal-trixie`

### 标签对照

| 标签 | 说明 |
|------|------|
| `17-minimal-trixie` | PostgreSQL 17 minimal 镜像（Debian trixie） |
| `17-system-trixie` | PostgreSQL 17 系统镜像（含额外工具） |
| `17.5-minimal-trixie` | 固定到 17.5 小版本 |

建议使用 `17-minimal-trixie` 跟踪最新的 17.x 小版本。

## 2. HA 模式与服务发现

### 说明

CNPG 自动创建以下 Service：
- `<cluster-name>-rw` — 指向主库（读写操作）
- `<cluster-name>-ro` — 指向从库（只读操作）
- `<cluster-name>-r` — 指向所有实例（负载均衡）

### 应用连接

应用应该连接 `-rw` 或 `-ro` Service，而不是连接外部 NodePort。
外部 NodePort 仅用于管理/开发调试。

## 3. 密码管理

### 说明

CNPG bootstrap 时从 `spec.bootstrap.initdb.secret` 读取初始密码。
之后 CNPG 自动管理密码并存储在 Kubernetes Secret 中。

### 获取密码

```bash
# 查看生成的用户密码
kubectl get secret pg-standalone-app -n pg -o jsonpath='{.data.password}' | base64 -d
kubectl get secret pg-standalone-superuser -n pg -o jsonpath='{.data.password}' | base64 -d
```

## 4. 节点维护操作

### 说明

CNPG 支持自动故障转移，但如果需要关停节点维护，
需要先切换主库到其他节点：

```bash
# 手动切换主库到指定节点
kubectl cnpg promote pg-ha pg-ha-2 -n pg
```

等待切换完成后再关停节点。
