# Skill 编写标准操作流程 (SOP)

本文档描述如何编写一个合格、可复用的 Skill。

## 什么是 Skill？

Skill 是**程序性记忆**，是可复用的工作流程和方法论，包含：
- 触发条件
- 具体操作步骤
- 常见问题与解决方案
- 验证步骤

## Skill 目录结构

一个标准的 Skill 包含：

```
your-skill-name/
├── SKILL.md              # 主文件（必需）
├── references/           # 参考文档（可选）
│   └── ...
├── templates/            # 模板文件（可选）
│   └── ...
├── scripts/              # 辅助脚本（可选）
│   └── ...
└── assets/               # 图片等资源（可选）
    └── ...
```

## SKILL.md 文件格式

SKILL.md 采用 YAML frontmatter + Markdown 正文格式：

```markdown
---

description: 简短描述这个 Skill 的用途
tags: [tag1, tag2]
related_skills: [other-skill-1, other-skill-2]
---
```

## 编写步骤

### 第一步：确定 Skill 范围

问自己以下问题：
- 这个 Skill 解决什么具体问题？
- 什么时候应该使用这个 Skill？
- 这个 Skill 有明确的开始和结束吗？

**原则**：一个 Skill 应该专注于做好一件事。

### 第二步：收集现有知识

- 整理已有的操作步骤
- 记录常见陷阱和错误
- 收集成功案例
- 列出需要的工具和资源

### 第三步：结构化内容

按照 SKILL.md 格式组织内容：
1. **概述** - 用 1-2 句话说明 Skill 用途
2. **适用场景** - 列出何时使用/何时不使用
3. **前置条件** - 列出需要准备的东西
4. **使用流程** - 编号步骤，精确描述
5. **常见问题** - 预见可能出错的地方
6. **最佳实践** - 经验总结

### 第四步：编写正文

**编写要点**：
- 使用主动语态
- 步骤清晰，可执行
- 提供具体命令和代码示例
- 注明需要用户决策的地方
- 描述预期结果

### 第五步：审核与测试

检查清单：
- [ ] 所有步骤都可执行吗？
- [ ] 有没有遗漏的前置条件？
- [ ] 常见问题是否覆盖了主要痛点？
- [ ] 语言是否清晰简洁？
- [ ] 有没有参考资源链接？

## Skill 质量标准

### 合格的 Skill 应该具备：

✅ **完整性** - 有头有尾，形成闭环
✅ **可操作性** - 步骤具体，可直接执行
✅ **实用性** - 解决真实问题，有使用场景
✅ **健壮性** - 考虑异常情况和错误处理
✅ **可维护性** - 结构清晰，易于更新

### 避免的问题：

❌ 过于宽泛，什么都想解决
❌ 步骤模糊，需要猜测
❌ 没有验证方法
❌ 不考虑出错情况
❌ 只有理论，没有实践

## 提交 Checklist

提交新 Skill 前确认：

- [ ] Skill 放在正确的分类目录下
- [ ] SKILL.md 格式正确
- [ ] 有明确的适用场景
- [ ] 步骤可执行
- [ ] 包含常见问题
- [ ] Git 提交信息规范

## 更新与维护

Skill 不是一成不变的：
- 使用后发现问题及时更新
- 工具版本更新时同步检查
- 收集反馈持续改进
- 定期回顾和归档过时 Skill
