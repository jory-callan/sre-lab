# GitOps 仓库重构结论

> 讨论时间：2026-07-11
> 讨论背景：gitops-base 仓库目录结构 + namespace 策略设计

---

## 核心认知

### 1. GitOps 理念 vs ArgoCD 工具

- GitOps 理念（Git 作为唯一真相源、声明式、可复现）**是对的**
- ArgoCD 只是 GitOps 的一种实现，解决的是**组织问题**（多集群、多团队、合规审计）
- 单集群个人运维场景下，ArgoCD 的价值几乎为零，反而引入了额外复杂度
- `install.sh` + `README.md` 钉死版本 已经是 GitOps 的核心诉求

### 2. 国外最佳实践参考

| 仓库 | ⭐ | 顶层目录 |
|------|-----|---------|
| glasskube/gitops-template | 181 | `apps/` `bootstrap/` `packages/` |
| gruberdev/local-gitops | 123 | `apps/` `config/` `storage/` |

- 国外高星仓库**没有纯平铺结构**，也没有四层分离
- 分类很克制（2-3 个顶层目录），分类维度是**运维属性**而非"为了分类而分类"
- 没有统一标准，每个仓库根据场景调整

## 推荐目录结构

```
k8s/
├── bootstrap/               ← 引导脚本（不纳入 ArgoCD）
│   ├── install-argocd.sh
│   ├── install-crds.sh
│   └── apply-root.sh
│
├── platform/                ← 集群平台基础设施
│   ├── cert-manager/        → namespace: cert-manager
│   ├── ingress-nginx/       → namespace: ingress-nginx
│   ├── metallb/             → namespace: metallb-system
│   └── nfs/                 → namespace: storage
│
├── observability/           ← 可观测性（统一 namespace）
│   ├── victoria-metrics/    → namespace: monitoring
│   ├── victoria-logs/       → namespace: monitoring
│   └── grafana/             → namespace: monitoring
│
├── operators/               ← Operator 控制面（统一 namespace）
│   ├── cnpg-operator/       → namespace: operators
│   ├── redis-operator/      → namespace: operators
│   └── minio-operator/      → namespace: operators
│
├── databases/               ← 数据库实例（各自独立 namespace）
│   ├── postgresql/          → namespace: postgresql
│   ├── redis/               → namespace: redis
│   └── minio/               → namespace: minio
│
├── apps/                    ← 业务应用（各自独立 namespace）
│   ├── gitea/               → namespace: gitea
│   ├── kdebug/              → namespace: kdebug
│   └── kite/                → namespace: kite
│
├── lab/                     ← 选型测试（不纳入 ArgoCD）
├── docs/
├── AGENTS.md
└── README.md
```

### 顶层目录分类逻辑

| 目录 | 安全域 | 运维模式 | 变更频率 |
|------|--------|---------|---------|
| `platform/` | 集群级，影响全局 | 装一次几乎不改 | 季度/年 |
| `observability/` | 观测面，只读为主 | 统一升级 | 月度 |
| `operators/` | 控制面，集群级 RBAC | 统一升级 | 季度 |
| `databases/` | 数据面，最高安全 | 各自独立运维 | 按需 |
| `apps/` | 业务面，最低安全 | 频繁变更 | 周/日 |

### 组件目录模板

```
postgresql/
├── README.md           ← 版本、架构、运维命令
├── install.sh          ← 手动部署入口
├── uninstall.sh        ← 手动卸载入口
├── 踩坑记录.md
├── kustomization.yaml
├── values.yaml
├── application.yaml    ← ArgoCD 入口（可选，可删）
├── cluster.yaml        ← 实例 CR
└── monitoring.yaml     ← ServiceMonitor / PrometheusRule
```

## Namespace 策略

### 生产最佳实践

| 类型 | namespace 策略 | 包含组件 |
|------|--------------|---------|
| **Operator 控制面** | 统一 `operators` | cnpg-operator, redis-operator, minio-operator |
| **平台基础设施** | 官方固定 namespace | cert-manager, ingress-nginx, metallb-system |
| **可观测性** | 统一 `monitoring` | VictoriaMetrics, VictoriaLogs, Grafana |
| **数据库/中间件** | **每个组件独立** | postgresql, redis, minio |
| **业务应用** | **每个应用独立** | gitea, kdebug, kite |

### 决策依据

- **控制面、观测面**：不对外暴露端口、不存业务数据 → 统一 namespace
- **数据库、业务应用**：存数据、对外暴露端口 → 各自独立 namespace

### 数据库独立 namespace 的理由

| 理由 | 说明 |
|------|------|
| **安全域** | 数据库是最高价值资产，不应和任何其他服务共享网络边界 |
| **备份策略不同** | PG 要 WAL 归档，Redis 要 RDB/AOF，MinIO 要 bucket 复制 |
| **QoS 保障** | 数据库通常 Guaranteed QoS，独立 ResourceQuota 才能精确保障 |
| **故障域** | Redis OOM 不应该影响 PostgreSQL 的故障排查 |
| **运维工具** | `kubectl get pods -n postgresql` 只看 PG，不用 grep |

## Namespace 常见误区

| 说法 | 真相 |
|------|------|
| "namespace 太多影响 etcd 性能" | etcd 压力来自资源数量（Pod/Secret），不是 namespace 数量 |
| "namespace 太多拖慢 API Server" | watch 连接数与 namespace 数量无关 |
| "namespace 太多影响 DNS 解析" | CoreDNS 压力来自 Service 数量，不是 namespace 数量 |
| "namespace 太多增加网络延迟" | namespace 不影响网络数据面 |

### 实际成本

| namespace 数量 | 体验 |
|---------------|------|
| 1-10 | 正常，`-n` 敲得过来 |
| 10-30 | 开始依赖 `-A` 和 kubectx/kubens |
| 30-100 | 需要工具辅助 |
| 100+ | 需要平台工程团队专门管理 |

**结论：namespace 没有性能问题，只有操作成本。10 个以内操作成本几乎为零。**

## 和旧 gitops-base 的对比

| 方面 | 旧方案（gitops-base） | 新方案 |
|------|---------------------|--------|
| 目录层数 | 4 层（infra/operators/middleware/apps） | 2 层（功能分类/组件名） |
| Application CRD | 集中在 argocd/ | 放在组件目录里 |
| 手动部署 | ❌ 没有入口 | ✅ install.sh |
| 卸载 | ❌ 靠 ArgoCD prune | ✅ uninstall.sh |
| 踩坑记录 | ❌ 没有 | ✅ 每个组件都有 |
| 删 ArgoCD 影响 | 需要重构整个仓库 | 删 application.yaml 即可 |
| Namespace 策略 | 共享 namespace 统一管理 | 按安全域独立/统一混合策略 |
